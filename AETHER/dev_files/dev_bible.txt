================================================================
  1. O QUE É O AETHER
================================================================

AETHER é um editor de texto local para escrita criativa.
Inspirado no Ellipsus e no Scrivener, mas completamente offline,
sem telemetria e sem qualquer servidor externo.

Nome completo : AETHER — Forja de Mundos
Versão atual  : 0.1.0
Última atualização: Fase 5 concluída (itens 5.1–5.6)
Plataformas   : Windows 10 e Linux (CachyOS / qualquer distro)

O usuário escolhe uma pasta no próprio computador (o "vault"),
e todos os dados vivem lá. A ferramenta nunca acessa a internet
a não ser para carregar as fontes do Google Fonts no momento
do desenvolvimento (em produção, as fontes devem ser bundladas).


================================================================
  2. CONTEXTO: ECOSSISTEMA OGMA
================================================================

O AETHER é a quarta ferramenta de um ecossistema pessoal:

  OGMA       — [ferramenta 1, não documentada aqui]
  KOSMOS     — [ferramenta 2, não documentada aqui]
  MNEMOSYNE  — [ferramenta 3, não documentada aqui]
  AETHER     — editor de escrita criativa (este projeto)

Todas as ferramentas compartilham o "OGMA Design Bible": mesma
paleta de cores sépia, mesma tipografia, mesmas regras de sombra
e animações base. O arquivo de design tokens (tokens.css) é a
implementação direta do Design Bible para o AETHER.

Diferenciais visuais do AETHER dentro do ecossistema:
  - Animação pageFloat: folhas de papel caem ao abrir capítulos
  - Animação pageExit: folha "voa" ao deletar
  - Efeito typewriter no splash e em loading states
  - Cursor do editor como _ piscante (não |)
  - CosmosLayer com labels mitológicos nas constelações
  - Nebulosas com pulso lento animado (etherPulse, 8s)

================================================================
  4. COMO RODAR O PROJETO
================================================================

PRÉ-REQUISITOS
  - Rust toolchain (rustup): https://rustup.rs
  - Node.js 18+ e npm
  - Tauri CLI: cargo install tauri-cli --version "^2"
  - Linux: libwebkit2gtk, libappindicator-gtk3, etc.
    (sudo pacman -S --needed webkit2gtk-4.1 libappindicator-gtk3)

DESENVOLVIMENTO
  # Na raiz do projeto:
  npm install          # instala dependências JS (só na primeira vez)
  cargo tauri dev      # inicia Tauri + Vite em modo dev

  O Vite roda em http://localhost:5173 e o Tauri abre a janela
  apontando para ele. Hot reload funciona no frontend.
  Alterações no Rust recompilam e reiniciam o processo backend.

BUILD DE PRODUÇÃO
  cargo tauri build    # gera binário + instalador em src-tauri/target/

APENAS O FRONTEND (sem Tauri)
  npm run dev          # Vite na porta 5173, sem backend Rust
  npm run build        # build estático para src-tauri/../dist/

================================================================
  6. ARQUITETURA GERAL
================================================================

FLUXO GERAL
  [Usuário] → [Janela Tauri (WebView)] → [React/TS] → [invoke()] → [Rust]
                                                                      ↓
                                                              [Sistema de arquivos]

O frontend é uma SPA React rodando dentro de uma WebView Tauri.
A comunicação com o backend Rust acontece via `invoke()` do
pacote @tauri-apps/api. Cada chamada retorna uma Promise.

COMUNICAÇÃO JS ↔ RUST
  - No Rust: funções decoradas com #[tauri::command]
  - No TS: invoke<ReturnType>('command_name', { args })
  - Todo command Rust retorna Result<T, AppError>
  - No TS, o tipo é: { ok: true; data: T } | { ok: false; error: AppError }
    (implementar wrapper em src/types/ — ver item 1.5 do todo)

ESTADO GLOBAL
  - Tema: classe 'dark' no <html>, persistido em localStorage
  - Vault path: Mutex<Option<PathBuf>> no AppState do Rust
  - Estado da UI: ainda não implementado (fase 1.6+)

TEMA DIA/NOITE
  Aplicado ANTES do primeiro render pelo initTheme() em main.tsx.
  Isso evita o "flash" de tema errado que ocorre quando o tema
  é aplicado após o React montar. O mecanismo é:
    1. main.tsx chama initTheme() (lê localStorage, adiciona class)
    2. Depois monta o React
  Para alternar: importar toggleTheme() de src/lib/theme.ts


================================================================
  7. MODELO DE DADOS E ARMAZENAMENTO
================================================================

PRINCÍPIO FUNDAMENTAL (modelo Obsidian)
  O usuário tem controle total sobre onde seus dados ficam.
  Cada vault é uma pasta normal — pode ser movida, copiada,
  versionada com git, sincronizada com qualquer ferramenta.
  O AETHER nunca armazena dados do usuário em servidores.

DOIS NÍVEIS DE ARMAZENAMENTO

  Nível 1 — AppData do sistema (gerenciado pelo Tauri)
    Linux  : ~/.local/share/aether/app.json
    Windows: %AppData%\aether\app.json

    Conteúdo de app.json:
    {
      "vault_path": "/caminho/escolhido/pelo/usuario"  // ou null
    }

    Isso é TUDO que fica no AppData. Somente o caminho do vault.
    Nenhum dado de conteúdo, nenhuma preferência do usuário.

  Nível 2 — Dentro do vault (portátil, do usuário)
    {vault}/
    ├── .aether/
    │   └── config.json          — preferências (tema, fonte, etc.)
    │
    └── {project_id}/            — uma pasta por projeto (UUID v4)
        ├── project.json         — metadados do projeto
        └── {book_id}/           — uma pasta por livro (UUID v4)
            ├── book.json        — metadados + lista de capítulos
            └── {chapter_id}.md  — conteúdo do capítulo (Markdown puro)

TIPOS DE DADOS (definidos em src-tauri/src/types.rs)

  AppData
    vault_path: Option<PathBuf>

  VaultConfig  (em .aether/config.json)
    theme:        Theme (Day | Dark)
    font_size:    u8    (padrão: 16)
    line_height:  f32   (padrão: 1.75)
    column_width: u16   (padrão: 680)

  Project  (= ProjectMeta, em project.json)
    id:              String (UUID v4)
    name:            String
    description:     Option<String>
    project_type:    ProjectType (Single | Series | Fanfiction)
    default_book_id: Option<String>  — só para Single; ID do livro único
    created_at:      String (ISO 8601)
    updated_at:      String (ISO 8601)
    -- metadados opcionais (todos com #[serde(default)]) --
    subtitle:        Option<String>
    genre:           Option<String>
    target_audience: Option<String>
    language:        Option<String>
    tags:            Vec<String>
    has_magic_system: bool
    tech_level:      Option<String>
    inspirations:    Option<String>

  Book  (em book.json)
    id:          String (UUID v4)
    project_id:  String
    name:        String
    order:       usize
    chapters:    Vec<ChapterMeta>
    series_name: Option<String>  — para Fanfiction: agrupa livros por série

  BookMeta  (versão compacta para listagens)
    id:          String
    name:        String
    order:       usize
    series_name: Option<String>

  ChapterMeta  (dentro de book.json — NÃO tem conteúdo)
    id:            String (UUID v4)
    title:         String
    order:         usize
    status:        ChapterStatus (Draft | Revision | Final)
    synopsis:      Option<String>
    word_count:    usize         (cache, recalculado ao salvar)
    trashed_at:    Option<String> (ISO 8601; None = ativo; Some = na lixeira)
    character_ids: Vec<String>   (4.6 — personagens vinculados)
    note_ids:      Vec<String>   (4.6 — notas de worldbuilding vinculadas)

  O conteúdo do capítulo é lido separadamente do arquivo .md.
  O scratchpad do capítulo fica em {chapter_id}.scratch.md (mesmo dir).

  Character  (em {project_id}/characters/{char_id}.json)
    id:           String (UUID v4)
    project_id:   String
    name:         String
    role:         Option<String>
    description:  Option<String>
    fields:       Vec<CustomField>   (campos livres label+value)
    image_path:   Option<String>     (caminho relativo ao vault)
    chapter_ids:  Vec<String>        (capítulos vinculados)
    created_at:   String (ISO 8601)
    updated_at:   String (ISO 8601)

  CustomField
    label:  String
    value:  String

  Relationship  (em {project_id}/characters/relationships.json — Vec)
    id:      String (UUID v4)
    from_id: String
    to_id:   String
    kind:    String   (ex: "Aliados", "Inimigos", "Família")
    note:    Option<String>

  WorldNote  (em {project_id}/worldbuilding/{note_id}.json)
    id:          String (UUID v4)
    project_id:  String
    name:        String
    category:    WorldCategory (location | faction | object | concept | other)
    description: Option<String>
    fields:      Vec<CustomField>
    image_path:  Option<String>
    chapter_ids: Vec<String>
    created_at:  String (ISO 8601)
    updated_at:  String (ISO 8601)

  TimelineEvent  (em {project_id}/timeline.json — Vec<TimelineEvent>)
    id:            String (UUID v4)
    project_id:    String
    title:         String
    description:   Option<String>
    date_label:    String   (rótulo livre: "Ano 1024", "Antes da guerra")
    order:         usize
    character_ids: Vec<String>
    note_ids:      Vec<String>
    chapter_ids:   Vec<String>

IDs
  Todos os IDs são UUID v4 gerados em Rust (uuid::Uuid::new_v4()).
  IDs são usados como nomes de pasta e de arquivo.
  O usuário nunca vê os IDs diretamente.

DATAS
  Todas as datas são strings ISO 8601 geradas com chrono::Utc::now().
  Não há timestamps Unix no modelo.


================================================================
  8. CAMADA RUST (src-tauri/src/)
================================================================

REGRA FUNDAMENTAL
  Toda função que pode falhar retorna Result<T, AppError>.
  .unwrap() e .expect() são PROIBIDOS em código de produção.
  Se uma função interna só pode falhar por bug (não por input
  do usuário), use um AppError descritivo mesmo assim.

── error.rs ──────────────────────────────────────────────────

  Define o enum AppError com as variantes:
    Io(String)              — erros de std::io
    Json(String)            — erros de serde_json
    VaultNotConfigured      — vault ainda não selecionado
    VaultNotFound(String)   — pasta do vault não existe
    ProjectNotFound(String) — projeto não encontrado
    BookNotFound(String)    — livro não encontrado
    ChapterNotFound(String) — capítulo não encontrado
    InvalidPath(String)     — caminho malformado
    InvalidName(String)     — nome vazio ou inválido
    AlreadyExists(String)   — ID já existe (colisão extremamente rara)

  AppError implementa:
    - thiserror::Error     (para Display e std::error::Error)
    - serde::Serialize     (obrigatório para retornar em commands Tauri)
    - From<std::io::Error> (conversão automática com ?)
    - From<serde_json::Error> (conversão automática com ?)

  O serde usa tag = "kind" + content = "message" para que o JSON
  de erro no frontend seja: { "kind": "ProjectNotFound", "message": "..." }

── types.rs ──────────────────────────────────────────────────

  Todos os structs e enums que são serializados para disco ou
  enviados ao frontend. Ver seção 7 para detalhes dos tipos.

  Regra: tipos que vão para o frontend derivam Serialize.
         Tipos que vêm do frontend derivam Deserialize.
         Tipos que vão para disco derivam ambos.

── lib.rs ────────────────────────────────────────────────────

  Entry point da biblioteca. Declara os módulos, define AppState

── components/CosmosLayer.tsx ────────────────────────────────

  SVG procedural determinístico. Mesmo seed = mesmo cosmos.
  Usado como fundo decorativo em cards, headers e splash.

  Props:
    seed:      number    — determina todo o layout do cosmos
    density?:  Density   — 'low' | 'medium' | 'high'
    animated?: boolean   — ativa/desativa etherPulse
    width?:    number    — largura do SVG (padrão: 520)
    height?:   number    — altura do SVG (padrão: 340)
    className?: string

  Densidades:

    medium — 2 nebulosas, 14 estrelas, cometa, lua (headers de livro)
    high   — 3 nebulosas, 19 estrelas, cometa, lua, planeta (splash)

  PRNG: mulberry32 inicializado com seed.
    Todas as posições e tamanhos derivam deste seed.
    Garante que o mesmo projeto sempre tenha o mesmo cosmos.

  Diferenciais AETHER:
    - Labels mitológicos nas constelações (Special Elite, 7px)
    - Nebulosas com animation: etherPulse 8s (quando animated=true)

  Como usar:
    <CosmosLayer seed={project.id_hash} density="medium" animated={true} />
    (Para converter project_id (string) em seed numérico, usar uma
    função de hash simples — ainda não implementada, fazer quando

── lib/theme.ts ──────────────────────────────────────────────

  Três temas: sépia (padrão), claro, escuro.
    sépia  — sem classe no <html> (padrão da folha de estilos)
    claro  — html.light  (fundo quase branco, tinta neutra)
    escuro — html.dark   (fundo sépia escuro, dourado brilhante)

  Exports:
    Theme           — 'sepia' | 'light' | 'dark'
    THEME_LABEL     — { sepia: 'Sépia', light: 'Claro', dark: 'Escuro' }
    initTheme()     — aplica tema salvo (chamar antes do render)
    getTheme()      — lê tema atual do DOM
    applyTheme(t)   — aplica e persiste
    cycleTheme()    — avança para o próximo tema e persiste

  Mecanismo: classe 'light' ou 'dark' no <html>. Sépia = sem classe.
  Persistência: localStorage, chave 'aether_theme'.

── lib/typography.ts ─────────────────────────────────────────

    DEFAULTS           — { fontSize: 16, lineHeight: 1.75, columnWidth: 680 }
    initTypography()   — lê localStorage, aplica CSS vars (chamar antes do render)
    applyTypography(s) — escreve --editor-font-size, --editor-line-height,
                         --editor-col-width em :root; persiste em localStorage

  Chave localStorage: 'aether_typography'.

── lib/searchReplace.ts ──────────────────────────────────────

  Extensão TipTap (Extension.create) que adiciona busca e substituição
  ao editor via plugin ProseMirror.

  SearchPluginState: { term, caseSensitive, matches[], current }
  searchKey: PluginKey — para ler o estado do plugin de fora.

  Decorations:
    .search-highlight         — todas as ocorrências (âmbar suave)
    .search-highlight-current — ocorrência atual (âmbar forte + outline)

  Commands adicionados ao editor:
    setSearchTerm(term, caseSensitive?) — busca e define matches
    goToMatch(index)          — navega para uma ocorrência específica
    replaceCurrentMatch(text) — substitui a ocorrência atual
    replaceAllMatches(text)   — substitui todas (do final para o início)
    clearSearch()             — limpa highlights e estado

  findMatches(doc, term, caseSensitive) — função pura auxiliar.
  Percorre doc.descendants() para encontrar posições exatas.

================================================================
  11. PADRÕES OBRIGATÓRIOS DE CÓDIGO
================================================================

Estes padrões não são sugestões. Toda PR ou commit que violar
qualquer um deles deve ser corrigido antes de ser aceito.

1. TRATAMENTO DE ERROS COM TIPAGEM É PRIORIDADE ABSOLUTA
   - Rust: toda função que pode falhar retorna Result<T, AppError>.
     Nunca .unwrap() ou .expect() em código de produção.
   - TypeScript: strict: true sempre.
     Erros de commands Tauri tipados com union type:
       type Result<T> = { ok: true; data: T } | { ok: false; error: AppError }
   - Erros tratados no ponto onde ocorrem — sem silenciar.

2. COMMIT APÓS CADA ITEM INDIVIDUAL DO TODO
   - Mensagem referencia o item: feat(fase-1): 1.3 comandos de projeto
   - Atualizar o status do item no todo ([x]) ANTES de commitar.

3. PRIVACIDADE É ABSOLUTA
   - Zero telemetria, zero analytics, zero conexões externas.
   - AppData do sistema contém APENAS o caminho do vault.
   - Todo dado do usuário fica dentro do vault escolhido por ele.
   - Não adicionar nenhuma dependência que faça chamadas de rede.

4. MODELO OBSIDIAN
   - Configurações em {vault}/.aether/ — nunca em ~/.aether/ global.
   - Cada projeto é uma pasta auto-contida e portátil.
   - O usuário pode mover/copiar/versionar o vault livremente.

5. CORES SEMPRE VIA VARIÁVEIS CSS
   - Nunca hardcodar cores em TSX/JSX/JS.
   - Exceção: o próprio tokens.css e os valores de STYLES no Toast.tsx
     (que existem porque aquelas cores específicas precisam de opacity
     que as variáveis CSS ainda não suportam diretamente).

6. FONTES CORRETAS POR CONTEXTO
   - IM Fell English: títulos e conteúdo, sempre itálico
   - Special Elite: UI, botões, labels
   - Courier Prime: blocos de código no editor

7. MANTER DEV_BIBLE.TXT ATUALIZADO
   - Ao concluir qualquer item de fase que introduza novos arquivos,
     módulos, comandos ou padrões, atualizar este documento.
   - O dev_bible deve refletir o estado atual do projeto, não o
     estado planejado.
