1. VISÃO GERAL DO PROJETO
================================================================================

Nome: OGMA
Inspiração: Ogma, deus celta da escrita, eloquência e conhecimento.

Conceito:
  Gerenciador e organizador unificado de projetos pessoais, estudos acadêmicos
  e leituras. Substitui e absorve completamente o StudyFlow. Inspirado no Notion,
  com estética de papel envelhecido / máquina de escrever / cosmos desenhado a
  lápis sobre pergaminho. 100% local, sem sincronização, sem conta, sem telemetria.
  Todos os dados armazenados exclusivamente em ogma/data/.

Equivalência com o StudyFlow:
  O StudyFlow chamava a entidade principal de "grade curricular".
  No OGMA, essa mesma entidade é um "Projeto do tipo Acadêmico".
  São a mesma coisa — apenas a terminologia foi unificada.
  Toda a lógica especial do StudyFlow (semestres, disciplinas, pré-requisitos,
  cores derivadas, fontes, áreas de conhecimento) é preservada integralmente
  dentro do tipo Acadêmico. Ao migrar dados do StudyFlow, cada grade vira
  um Projeto Acadêmico e cada disciplina vira uma Página do tipo Disciplina.

Módulo de Leituras:
  Faz parte da Biblioteca de Recursos — não é um módulo separado.
  Na sidebar, "Biblioteca" expande em duas subseções: Recursos e Leituras.
  Um livro existe uma única vez na Biblioteca. Páginas e disciplinas de qualquer
  projeto podem referenciar esse livro via vínculo (tabela reading_links).

Plataformas-alvo:
  - CachyOS / Arch Linux (plataforma principal de desenvolvimento)
    Usuário: spacewitch — pasta do projeto em ~/Documents/OGMA/
  - Windows 10 x64

================================================================================
6. API IPC — REFERÊNCIA COMPLETA (window.db)
================================================================================

A bridge é definida em src/main/preload.ts e exposta como window.db no renderer.
Todos os métodos retornam Promises (via ipcRenderer.invoke).
Todos os handlers IPC são síncronos no lado main (better-sqlite3 é síncrono).

O wrapper api() em ipc.ts encapsula cada handler e retorna sempre:
  { ok: true, data: resultado }   — sucesso
  { ok: false, error: 'mensagem' } — erro capturado

CONVENÇÃO DE NOMENCLATURA:
  Canal IPC:    'tabela:operacao'         ex: 'projects:create'
  Acesso React: window.db.tabela.operacao ex: window.db.projects.create(data)

────────────────────────────────────────────────────────────────────────────────
window.db.workspace
────────────────────────────────────────────────────────────────────────────────
  .get()
    → Workspace   — o workspace único do app

  .update(data)
    → Workspace
    data: { name?, icon?, accent_color? }

────────────────────────────────────────────────────────────────────────────────
window.db.projects
────────────────────────────────────────────────────────────────────────────────
  .list()
    → Project[]   — todos os projetos do workspace

  .create(data)
    → Project
    data: { name, project_type, description?, icon?, color?, subcategory?,
            date_start?, date_end?, extra_fields? }

  .update(data)
    → Project
    data: { id, ...campos a atualizar }

  .delete({ id })
    → { ok: true }   — CASCADE deleta todas as páginas do projeto

────────────────────────────────────────────────────────────────────────────────
window.db.pages
────────────────────────────────────────────────────────────────────────────────
  .list({ project_id })
    → Page[]   — páginas do projeto (não deletadas), ordenadas por sort_order

  .create(data)
    → Page
    data: { project_id, title, page_type, icon?, content_json?, sort_order? }

  .update(data)
    → Page
    data: { id, ...campos a atualizar }

  .delete({ id })
    → { ok: true }   — soft delete (is_deleted = 1)

────────────────────────────────────────────────────────────────────────────────
window.db.config
────────────────────────────────────────────────────────────────────────────────
  .get({ key })      → string | null

8. EDITOR DE TEXTO RICO — COMO FUNCIONA
================================================================================

  O Editor.js é carregado via dynamic import diretamente no renderer React,
  montado numa <div> via ref. Esta é a mesma abordagem usada pelo StudyFlow
  (que serviu de referência para esta solução).

  NÃO usa:
    - iframe
    - QWebChannel / QWebEngineView (era a abordagem da versão Python — abandonada)
    - Bundle separado (era o EditorFrame antigo — também abandonado)
    - Build step separado do editor

  Como funciona (EditorFrame.tsx):

    const holderRef = useRef<HTMLDivElement>(null)
    const editorRef = useRef<any>(null)

    useEffect(() => {
      const init = async () => {
        const [
          { default: EditorJS },
          { default: Header },
          // ... outros plugins
        ] = await Promise.all([
          import('@editorjs/editorjs'),
          import('@editorjs/header'),
          // ...
        ])

        editorRef.current = new EditorJS({
          holder: holderRef.current,
          data: JSON.parse(content ?? '{}'),
          tools: { header: { class: Header, ... }, ... },
          onChange: async () => {
            const output = await editorRef.current.save()
            onSave(JSON.stringify(output))  // debounce 1.5s
          },
          onReady: () => setLoading(false),
        })
      }
      init()
      return () => { editorRef.current?.destroy() }
    }, []) // monta uma vez

    return <div ref={holderRef} className="editor-holder" />

  Troca de conteúdo sem recriar o editor:
    Quando a prop `content` muda (usuário navega para outra página),
    o EditorFrame chama editorRef.current.render(newData) ao invés de
    recriar a instância. Isso é mais rápido e evita flicker.

  Auto-save:
    onChange dispara um setTimeout de 1500ms. Se o usuário continuar
    digitando, o timer é reiniciado (clearTimeout). Isso garante que
    o save só acontece 1.5s após a última modificação.

  Formato salvo em pages.content_json:
    {
      "time": 1234567890,
      "blocks": [
        { "type": "header",    "data": { "text": "Título", "level": 2 } },
        { "type": "paragraph", "data": { "text": "Texto..." } },
        { "type": "list",      "data": { "style": "unordered", "items": ["a","b"] } }
      ],
      "version": "2.31.x"
    }

  Tema do editor:
    Aplicado via CSS scoped em .editor-holder (EditorFrame.css).
    O toggle de tema dark/light adiciona/remove a classe .editor-dark no holder.
    Sem !important em excesso — as variáveis CSS do design system são herdadas.

11. ARMADILHAS TÉCNICAS CONHECIDAS
================================================================================

1. DYNAMIC IMPORT NO EDITOR.JS
   O Editor.js e seus plugins devem ser importados com dynamic import DENTRO
   de funções assíncronas. Import estático no topo do arquivo quebra no Electron
   porque o Chromium do Electron processa os módulos de forma diferente.

2. TROCA DE CONTEÚDO NO EDITOR
   Quando o usuário navega para outra página, não destrua e recrie o editor.
   Use editorRef.current.render(newData). Recriar o editor causa flicker e
   é desnecessariamente custoso.

3. FOCO EM INPUTS DENTRO DE MODAIS (ELECTRON/WINDOWS)
   No Windows, usar onClick + stopPropagation no overlay de modal para fechar
   conflita com o foco em inputs dentro do modal. O padrão correto é:
     onMouseDown={e => { if (e.target === e.currentTarget) onClose() }}
   Isso só fecha o modal quando o clique é no overlay, não no conteúdo.

4. DATAS ISO E FUSO HORÁRIO
   Ao exibir datas ISO (YYYY-MM-DD) como datas locais, sempre adicione T00:00:00:
     new Date('2026-03-21T00:00:00')  ← correto
     new Date('2026-03-21')           ← pode mostrar dia anterior em GMT-3

5. BETTER-SQLITE3 E GCC 15
   No CachyOS (Arch) com GCC 15, o better-sqlite3 padrão falha no rebuild
   devido a mudanças no padrão C++20. Usar SEMPRE o pacote
   better-sqlite3-multiple-ciphers que tem binários pré-compilados.

6. .npmrc NA RAIZ
   O arquivo .npmrc pode conter a linha:
     better_sqlite3_multiple_ciphers_binary_host_mirror=...
   que gera avisos durante o npm install. Pode ser removido com segurança
   — não afeta a instalação. O .gitignore já exclui o .npmrc.

7. CONCURRENTLY E ENCERRAMENTO
   O script dev usa --kill-others no concurrently para garantir que tanto
   o Vite quanto o Electron sejam encerrados quando qualquer um dos dois sair.
   Sem isso, o processo Vite pode continuar rodando em background.

8. PRELOAD E SANDBOX
   O preload.ts requer sandbox: false no BrowserWindow para funcionar
   corretamente com ipcRenderer. Com sandbox: true, o preload não tem acesso
   ao ipcRenderer.

15. MÓDULO ACADÊMICO — HERANÇA CRÍTICA DO STUDYFLOW
================================================================================

O StudyFlow era um app standalone de organização acadêmica.
O OGMA absorve 100% das suas funcionalidades dentro do tipo de projeto "academic".

Mapeamento de terminologia:
  StudyFlow               OGMA
  ────────────────────────────────────
  Grade curricular    →   Projeto (project_type = 'academic')
  Disciplina          →   Página (page_type = 'discipline')
  Semestre            →   Seção dentro do projeto acadêmico
  Fonte               →   academic_sources (a implementar)
  Área de conhecimento→   academic_areas (a implementar)

Funcionalidades específicas do módulo acadêmico (Fase 6):

  CORES DERIVADAS (colorUtils.ts):
    Cada disciplina tem uma cor base. A partir dela, o sistema HSL gera
    9 variações tonais (mais claro → mais escuro). A cor é usada na
    borda esquerda do card, no header da página e na aba ativa.

  GERADOR DE CÓDIGO:
    IPC: disciplinas:gerarCodigo({ nome, projectId })
    Lógica: extrai iniciais das palavras principais (sem stopwords PT-BR),
    forma um prefixo de até 4 chars, adiciona número sequencial de 3 dígitos
    para evitar colisão (ex: 'MAT001', 'CALC002').

  PRÉ-REQUISITOS:
    Detecção de ciclo via BFS antes de salvar.
    Semântica: "A requer B" = B deve ser concluída antes de iniciar A.
    Ao tentar adicionar A→B, verifica se B→A já existe.

  PROGRESSO:
    Calculado em tempo real: disciplinas concluídas / total.
    Por semestre e por projeto inteiro.

16. MIGRAÇÃO DE DADOS DO STUDYFLOW
================================================================================

Script futuro: src/main/studyflow_migration.ts
Acesso: Configurações → "Importar dados do StudyFlow"
Idempotente (pode ser executado múltiplas vezes com segurança).

MAPEAMENTO PRINCIPAL:
  grades              → projects (project_type='academic')
  semestres           → seções dentro do projeto (a definir schema)
  disciplinas         → pages (page_type='discipline')
  fontes              → academic_sources
  areas               → academic_areas
  disciplina_areas    → academic_discipline_areas
  prerequisitos       → academic_prerequisites
  tarefas             → academic_tasks
  prazos              → academic_deadlines
  recursos            → resources (Biblioteca)
  eventos_calendario  → calendar_events
  sessoes_pomodoro    → time_sessions
  configuracoes       → config

CAMPOS PRESERVADOS:
  disciplina.conteudo (JSON do Editor.js do StudyFlow)
    → Compatível diretamente com o Editor.js do OGMA.
    Não precisa de transformação.

  disciplina.cor, disciplina.banner_url, etc.
    → Mapeados para os campos equivalentes da tabela pages + page_properties.
