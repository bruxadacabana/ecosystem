1. VISAO GERAL E MODELO MENTAL
================================================================================

Nome: OGMA
Inspiracao: Ogma, deus celta da escrita, eloquencia e conhecimento.

Conceito:
  Gerenciador e organizador unificado de projetos pessoais, estudos academicos
  e leituras. Substitui e absorve completamente o StudyFlow. Inspirado no Notion,
  com estetica de papel envelhecido / maquina de escrever / cosmos desenhado a
  lapis sobre pergaminho. 100% local, sem sincronizacao, sem conta, sem telemetria.
  Todos os dados armazenados exclusivamente em ogma/data/.

────────────────────────────────────────────────────────────────────────────────
MODELO MENTAL FUNDAMENTAL (ler com atencao)
────────────────────────────────────────────────────────────────────────────────

  PROJETO = banco de dados de paginas (como uma Database do Notion)
  PAGINA  = a unidade basica de dados. Tem titulo + propriedades + corpo de texto.
  VIEW    = uma forma de visualizar as paginas de um projeto.

  Kanban, Tabela, Calendario e Lista NAO sao tipos de pagina.
  Sao formas diferentes de visualizar o mesmo conjunto de paginas.

  Exemplo concreto:
    Projeto "TCC" tem paginas: "Introducao", "Metodologia", "Resultados"
    Cada pagina tem propriedades: Status, Data Limite, Prioridade
    View Kanban:     mostra as paginas agrupadas por Status (colunas)
    View Tabela:     mostra as paginas como linhas com todas as propriedades
    View Calendario: mostra as paginas posicionadas por Data Limite
    Os mesmos dados, apresentados de formas diferentes.

  Uma pagina sempre tem:
    - Titulo (campo fixo, sempre presente)
    - Propriedades configuráveis (definidas pelo projeto)
    - Corpo de texto rico (Editor.js -- pode estar vazio)

  Abrir uma pagina (de qualquer view) mostra:
    ┌──────────────────────────────────────────┐
    │ [icone] Titulo da Pagina                 │
    │ [capa]                                   │
    ├──────────────────────────────────────────┤
    │ Status:      Em andamento                │
    │ Prioridade:  Alta                        │
    │ Data Limite: 01/04/2026                  │
    ├──────────────────────────────────────────┤
    │ Corpo (Editor.js)                        │
    │ Escreva algo...                          │
    └──────────────────────────────────────────┘

────────────────────────────────────────────────────────────────────────────────
EQUIVALENCIA COM O STUDYFLOW
────────────────────────────────────────────────────────────────────────────────
  Grade curricular  ->  Projeto (project_type = 'academic')
  Disciplina        ->  Pagina (com propriedades: Semestre, Area, Status, etc.)
  Semestre          ->  Propriedade do tipo select dentro do projeto academico
  Fonte             ->  Propriedade do tipo select ou text
  Area              ->  Propriedade do tipo multi_select

────────────────────────────────────────────────────────────────────────────────
MODULO DE LEITURAS
────────────────────────────────────────────────────────────────────────────────
  Faz parte da Biblioteca -- nao e um modulo separado.
  Sidebar: Biblioteca expande em Recursos e Leituras.
  Um livro existe uma vez na Biblioteca e pode ser vinculado a multiplas paginas
  via reading_links.

4. HIERARQUIA DE DADOS -- MODELO NOTION
================================================================================

  WORKSPACE (1)
    └── PROJETO (N)
          ├── project_properties  -- define os campos das paginas deste projeto
          ├── project_views       -- formas de visualizar as paginas
          └── PAGINAS (N)
                ├── title         -- campo fixo (sempre presente)
                ├── icon, cover   -- campos fixos opcionais
                ├── body_json     -- corpo Editor.js (pode ser vazio)
                └── page_prop_values -- valores das propriedades do projeto

  RELACIONAMENTOS ADICIONAIS:
    Pagina <-> Pagina:     page_backlinks (backlinks do editor)
    Pagina -> Tags globais: page_tags (para busca cross-projeto)
    Leitura <-> Pagina:    reading_links (Biblioteca)

  DECISOES ARQUITETURAIS FECHADAS:
    - Titulo: campo fixo da tabela pages. Nao e uma project_property.
    - Subpaginas: existem (parent_id). Visibilidade por view configuravel
      via include_subpages (0=so raiz, 1=todas). Padrao: 0.
    - Sub-databases: nao na v1. Cada pagina tem so corpo de texto rico.
    - Tags: dois sistemas paralelos -- tags globais (cross-projeto) e
      propriedade multi_select local ao projeto. Independentes.

6. PROPRIEDADES -- TIPOS E DEFAULTS POR TIPO DE PROJETO
================================================================================

Quando um projeto e criado, seedProjectProperties() cria automaticamente
as propriedades padrao para o tipo. O usuario pode adicionar, remover e
renomear depois (exceto is_built_in = 1).

TIPOS DE PROPRIEDADE SUPORTADOS:
  text         Texto curto livre (Professor, Codigo, URL)
  number       Numero (Carga horaria, Creditos, Progresso %)
  select       Uma opcao de lista -- as options definem colunas do Kanban
  multi_select Multiplas opcoes (Tags, Areas)
  date         Data ou intervalo (value_date + value_date2 opcional)
  checkbox     Sim/Nao
  url          Link externo
  color        Cor hex -- para cores de disciplinas academicas

  Futuros (nao implementar agora): relation, rollup, formula

PROPRIEDADE ESPECIAL STATUS:
  Todo projeto tem uma propriedade "Status" do tipo select.
  Os options do Status sao as colunas do Kanban.
  Sem Status configurado -> view Kanban indisponivel.

PROPRIEDADE ESPECIAL DATE:
  Projetos com propriedade do tipo date podem usar a view Calendario.
  O campo de data para o calendario e configuravel por view (date_property_id).

================================================================================
7. VIEWS -- TIPOS E COMPORTAMENTO
================================================================================

Views sao salvas em project_views. Um projeto pode ter multiplas views.

── TABLE (Tabela) ────────────────────────────────────────────────────────────
  Lista paginas como linhas. Colunas = propriedades visiveis (configuravel).
  Suporta: ordenacao por coluna, filtros, busca.
  Disponivel: sempre.

── KANBAN ────────────────────────────────────────────────────────────────────
  Agrupa paginas em colunas por uma propriedade select (group_by_property_id).
  Padrao: agrupar por "Status". Cada option = uma coluna.
  Cards mostram titulo + propriedades selecionadas.
  Mover card = UPDATE page_prop_values para o novo option_id.
  + coluna "Sem valor" para paginas sem valor definido.
  Disponivel: projetos com ao menos uma propriedade select.

── CALENDAR (Calendario) ─────────────────────────────────────────────────────
  Posiciona paginas em calendario mensal pelo valor de date_property_id.
  Mover no calendario = UPDATE page_prop_values para nova data.
  Disponivel: projetos com ao menos uma propriedade date.

── LIST (Lista) ──────────────────────────────────────────────────────────────
  Lista compacta. Suporta agrupamento, filtros, ordenacao.
  Disponivel: sempre.

── GALLERY (Galeria) ─────────────────────────────────────────────────────────
  Cards em grade com imagem de capa. Versao futura -- nao implementar agora.

SUBPAGINAS NAS VIEWS:
  Controlado por include_subpages na project_view.
  0 (padrao): WHERE parent_id IS NULL -- so paginas raiz.
  1: WHERE project_id = ? -- todas as paginas incluindo subpaginas.

SEED DE VIEWS POR TIPO:
  academic:  Tabela + Kanban (por Status) + Calendario (por Data Fim)
  software:  Kanban (por Status) + Tabela + Calendario (por Data Limite)
  health:    Lista + Calendario (por Data Inicio) + Tabela
  creative:  Kanban (por Status) + Galeria + Tabela
  research:  Tabela + Lista
  custom:    Tabela + Kanban (por Status)

9. SEED AUTOMATICO AO CRIAR PROJETO
================================================================================

Quando projects:create e chamado, o handler executa internamente:
  1. Insere o projeto
  2. seedProjectProperties(projectId, projectType)
     -> Cria as project_properties padrao para o tipo (ver secao 6)
     -> Para cada select/multi_select, cria as prop_options padrao
  3. seedProjectViews(projectId, projectType)
     -> Cria as views padrao para o tipo (ver secao 7)
     -> Define a primeira como is_default = 1
  4. Retorna o projeto com propriedades e views ja embutidas

Isso garante que todo projeto novo esta funcional sem configuracao adicional.

19. MODULO ACADEMICO -- HERANCA CRITICA DO STUDYFLOW
================================================================================

O StudyFlow era um app standalone academico. O OGMA absorve tudo dentro do
tipo de projeto 'academic'. Cada grade curricular vira um projeto academico,
cada disciplina vira uma pagina com as propriedades do modulo academico.

Funcionalidades especificas (Fase 6):

CORES DERIVADAS (colorUtils.ts):
  Propriedade "Cor" do tipo color em cada disciplina/pagina.
  A partir da cor base, o sistema HSL gera 9 variacoes tonais.
  Usada na borda esquerda do card na view e no header da pagina aberta.

GERADOR DE CODIGO (IPC: pages:generateCode):
  Input: { title, projectId }
  Logica: extrai iniciais das palavras principais (sem stopwords PT-BR),
  prefixo ate 4 chars, + numero sequencial 3 digitos evitando colisao.
  Ex: "Calculo I" -> "CALC001", "Matematica Discreta" -> "MATD001"

PRE-REQUISITOS:
  Tabela separada (a definir na Fase 6): academic_prerequisites.
  Deteccao de ciclo via BFS antes de salvar.
  "A requer B" = B deve ser concluida antes de iniciar A.

PROGRESSO:
  Calculado em tempo real: COUNT(*) WHERE Status = 'Concluida' / COUNT(*) total
  Por semestre (filtro na propriedade Semestre) e por projeto inteiro.

20. MIGRACAO DE DADOS DO STUDYFLOW
================================================================================

Script: src/main/studyflow_migration.ts (Fase 6)
Acesso: Configuracoes -> "Importar dados do StudyFlow"
Idempotente.

MAPEAMENTO PRINCIPAL:
  grades         -> projects (project_type='academic')
  disciplinas    -> pages + page_prop_values (Status, Semestre, Area, etc.)
  disciplina.conteudo -> pages.body_json  (JSON do Editor.js -- compativel direto)
  disciplina.cor -> page_prop_value para propriedade "Cor"
  semestres      -> prop_options da propriedade "Semestre"
  areas          -> prop_options da propriedade "Area"
  prerequisitos  -> academic_prerequisites
  tarefas        -> checklist no body_json ou tabela de tasks
  prazos         -> calendar_events vinculados a pagina
  recursos       -> resources (Biblioteca)
  eventos_cal.   -> calendar_events
  sessoes_pomodoro -> time_sessions
  configuracoes  -> config
